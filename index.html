Full Review

Architecture, modules, risk, and roadmap.

Technical path source: https://github.com/LUTAO581314/hermes-

You are reviewing the MOXI Agent System technical path.

Goal:
Build a lightweight personal and company agent system with conversation, tool calling, governed memory, public-opinion intelligence, image understanding, voice interaction, and later company workflow management.

Architecture:
- Lightweight VPS is only the orchestrator.
- Heavy capabilities are API-first or external-runtime-first.
- Interaction surfaces stay thin.
- Channel policy separates personal, company, admin, and high-risk actions.
- Orchestration core owns queues, tool calls, retries, and model selection.
- Model gateway exposes fast, summary, reasoning, and vision slots.
- Memory governor separates working memory from reviewed durable notes.
- Public-opinion intelligence keeps source and freshness metadata.
- Company workflow starts read-only and uses approval gates before writes.
- Sticker/media expression stores provider metadata or reviewed generation instructions, not bundled image files; channels that cannot upload images use explicit text fallback.
- Safety layer controls permissions, logs, secrets, and high-risk actions.

Please produce feasibility, risks, missing modules, optimized roadmap, memory governance improvements, public-opinion improvements, company-workflow hardening, performance improvements, and the next engineering task.

请帮我评审这个 MOXI 智能体系统技术路径。目标是在轻量服务器上运行一个个人和公司都能用的 Agent 系统。

核心路线:
用户消息 -> 交互入口 -> 渠道策略 -> 编排核心 -> 受治理的记忆上下文 -> 工具或外部运行时 -> 模型回复 -> 可见结果 -> 经过审核的记忆/报告写回。

关键原则:
1. 轻量服务器只做编排，不跑重型本地大模型。
2. 推理、视觉、搜索扩展、正式 TTS、未来视频理解优先走 API 或外部运行时。
3. 记忆不能乱存，工作记忆和长期记忆分离，长期记忆要经过审核和做梦整理。
4. 舆情热点要有来源、时间、新鲜度，不能把热榜当事实。
5. 公司管理先做只读，再做提醒和报告，写任务、日程、表格必须确认。
6. 交易、资金、审批、HR、法务、删除文件等高风险动作默认禁用。
7. BaiLongma 等上游项目作为外部运行时接入，MOXI 仓库保存技术路径、运行时契约、部署脚本、补丁和归因规则。

请输出可行性、最大风险、缺失模块、下一步三阶段路线，以及如何批量复制但保留仓库来源。

Dependency Strategy

MOXI keeps the main repository small and auditable. Upstream projects are installed as runtimes, then shaped by documented overlays.

MOXI Repository

Owns runtime contracts, CI, deployment scripts, memory governance, public copy pack, reports, and attribution.

BaiLongma Runtime

Runs as an external MIT-licensed Node/Electron checkout or fork. MOXI changes are tracked as overlays and patches.

Patch Boundary

Connector UI, QQ settings, latency improvements, voice/image bridge, and persona branding should become focused patch files.

CI Boundary

GitHub Actions checks this repository. Heavy Electron smoke tests stay in the BaiLongma checkout or future dedicated fork CI.

Secret Boundary

Server .env, API keys, QR sessions, chat ids, media files, and logs stay outside Git and are validated by hygiene checks.

Credit Boundary

External copies must keep the MOXI technical path source line while preserving upstream licenses for copied source.

Capability Matrix

The frontend adapter does not rewrite Hermes logic. It reads secret-safe readiness state, shows progress, and routes normalized channel turns into the runtime contract.

Runtime Core

`/health`, `/ready`, `/frontend/contract`, `/config/schema`, `/config/update`, `/social/turn`, `/jobs/event`, `/latency`, and `/capabilities` form the control-plane contract.

Personal Channels

WeChat, QQ official bot, NapCat-backed QQ personal scan, web chat, voice, stickers, and browser TTS remain warm personal surfaces with strict action limits.

Company Plane

Feishu owns company identity, group context, read-only data, approval-gated writes, and visible company-plane chat badges.

Hermes Logic

Hermes keeps native agent reasoning and tool execution. MOXI adapts surfaces and visibility around it.

Settings Control

Brain UI settings now operate as a Hermes control center with overview, capability matrix, frontend contract, writable config schema, runtime-config save UI, performance budget, memory status, async jobs, social channels, model, media, voice, search, security, appearance, and update domains.

Progress UI

BaiLongma now preflights `/message` through Hermes `/social/turn`, emits `moxi_progress`, reports native worker lifecycle events, merges active-job follow-ups, and shows route-aware progress plus plane badges while slow work runs.

Implementation Roadmap

Build the reliable core first. Company workflow, rich media, and risky automation come later with stronger gates.

1. Runtime

Service user, health checks, logs, restart policy, scripts, and secret hygiene.

2. Core Loop

Message normalization, queue, model gateway, tool contract, image reading, voice input.

3. Memory

Working memory, durable notes, review gates, dream consolidation, correction workflow.

4. Intelligence

Hot-list panel, source links, search expansion, deduplication, clustering, reports.

5. Company

Identity, group routing, idempotency, read-only Feishu user lookup and Bitable records, then approval-gated writes.

6. Media

Browser or provider TTS, metadata-only sticker bridge, reviewed runtime image generation, upload-or-fallback media sends.

Module Boundaries

Keep responsibilities narrow so one chat surface never controls the entire system.

Interaction Surface

Web chat, personal channel, company channel, voice controls, visual panels.

Channel Policy

Classifies personal, company, admin, and high-risk requests before execution.

Orchestration Core

Queueing, priority, retries, task locks, model selection, and tool routing.

Memory Governor

Admission rules, review, dream consolidation, durable write-back, and correction.

Intelligence Layer

Hot lists, search expansion, source freshness, clustering, risk scoring, reports.

Sticker Bridge

Prepared-sticker metadata, optional reviewed image generation, runtime upload, and explicit fallback when a channel cannot send images.

Safety And Audit

Permission levels, approval gates, logs, secret handling, rollback boundaries.
