# Phase 06 中文报告：社交响应与底层性能优化

## 1. 本阶段目标

本阶段目标是解决社交渠道“等很久才回复、不像真人”的问题，并把后续底层性能优化路线写入仓库，方便持续迭代和给外部 AI/同学复用。

核心原则：

- 先修表层体感，让微信、飞书不再沉默等待。
- 再修底层性能，把慢点变成可测量、可路由、可异步的工程问题。
- 文档公开可复制，但不暴露私有部署、服务器、密钥和聊天数据。

## 2. 已完成内容

### 2.1 表层方案固化

已将以下体验规则写入技术路径：

- 复杂问题先发“我想想哦，马上回你”。
- 读图先发“我看一下这张图，等我一下哦”。
- 生图先发“等我拍一下，马上给你”。
- 搜索先发“我查一下，别急”。
- ACK 只是进度提示，不算最终回答。
- 工具完成后必须继续补发最终结果。

### 2.2 底层性能配置

新增最小运行时性能配置：

- `HERMES_SOCIAL_QUICK_ACK_DELAY_MS`
- `HERMES_SOCIAL_FAST_REPLY_TARGET_MS`
- `HERMES_SLOW_TASK_THRESHOLD_MS`
- `HERMES_ASYNC_TASK_TIMEOUT_SECONDS`
- `HERMES_LATENCY_TELEMETRY_ENABLED`

这些配置用于定义社交快速反馈、5 秒快答目标、慢任务阈值、异步任务超时和耗时遥测开关。

### 2.3 性能、路由、延迟端点

新增安全运行时端点：

```text
GET /performance
GET /route?message=...
GET /latency
POST /latency/turn
```

这些端点可以返回：

- quick ack 延迟预算；
- 快答目标；
- 慢任务阈值；
- 异步任务超时；
- 模型路由槽位；
- 意图路由结果；
- 安全阶段耗时记录。

端点不会返回 API Key、密钥、服务器地址、聊天记录或私有运行细节。

## 3. 当前架构判断

当前慢感不是单纯模型能力问题，而是链路问题：

```text
入站消息
  -> 上下文构建
  -> 工具选择
  -> 模型首 token
  -> 慢工具
  -> 最终发送
```

因此优化重点是：

- 快速 ACK；
- 上下文瘦身；
- 工具按需加载；
- 慢任务异步化；
- 模型分层路由；
- 阶段耗时可观测。

## 4. 风险与约束

- ACK 不能泛滥，否则会像机器人刷屏。
- ACK 不能替代最终结果。
- 快模型不能处理高风险决策。
- 飞书公司写操作仍然必须经过主人确认。
- 生图、读图、搜索等 API 延迟仍受上游影响，必须异步化。
- 性能端点不能泄露任何密钥和私有部署信息。

## 5. 验收结果

本阶段验收通过：

- 仓库文档能清楚说明“先表层、再底层”的性能路线。
- `/performance` 能返回安全性能 profile。
- `/route` 能返回规则优先的路由结果。
- `/latency` 能查看安全耗时记录。
- 测试覆盖性能配置、路由和延迟端点。
- 未提交任何 API Key、服务器密码或私有聊天内容。
